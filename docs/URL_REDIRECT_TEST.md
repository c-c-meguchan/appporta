# 公開URL変更・リダイレクトの挙動テスト

## 前提

- マイグレーション `012_url_redirects.sql` を適用済みであること。
- 公開済みアプリが1つ以上あること。

## テスト1: 矢印が下向き＆注意文の表示

1. アプリの **設定** を開く。
2. 公開URLの **編集** でスラッグを変更し **保存**（例: `myapp` → `myapp-new`）。
3. ヘッダーの **更新** を押す。
4. 確認ダイアログで次を確認する：
   - 変更前→変更後の間に **下向き矢印（↓）** が表示されている。
   - 「変更すると旧URLは30日間新URLにリダイレクトされます。リダイレクトは直前のURLのみ保持され、それ以前のURLは無効になります。」の文が表示されている。

## テスト2: 旧URL→新URL のリダイレクト（30日間）

1. 公開済みアプリの **現在の公開URL** をメモする（例: `http://localhost:3000/old-slug`）。
2. **設定** で公開URLを変更し **保存**（例: `old-slug` → `new-slug`）。
3. ヘッダー **更新** を押し、ダイアログで **変更する** を実行。
4. ブラウザは新URLの設定画面に遷移する。
5. **別タブ** を開き、**旧URL**（`http://localhost:3000/old-slug`）にアクセスする。
6. **期待**: 旧URLにアクセスすると、自動的に **新URL**（`http://localhost:3000/new-slug`）にリダイレクトされ、公開ページが表示される。
7. アドレスバーのURLは新URLに変わっていること。

## テスト3: 「直前のURLのみ保持」の確認

1. 上記の状態（`old-slug` → `new-slug` に変更済み）から、さらに **設定** でURLを変更する（例: `new-slug` → `newest-slug`）。
2. **更新** → **変更する** で適用する。
3. 次の2つを確認する：
   - **新URL（new-slug）にアクセス** → **newest-slug** にリダイレクトされる（直前のみ保持）。
   - **旧URL（old-slug）にアクセス** → リダイレクトされず **Not Published**（または404）になる（それ以前のURLは無効）。

## テスト4: 期限切れ後の挙動（任意）

- Supabase の `url_redirects` テーブルで、該当行の `expires_at` を過去の日時に手動で更新する。
- 旧URLにアクセス → リダイレクトされず Not Published になることを確認する。
- 本番では30日経過後に同じ挙動になる。

## 補足

- リダイレクトは **公開ページ**（`/[appID]`）にアクセスしたときのみ動作する。管理画面（`/apps/[appID]/...`）のURLはリダイレクト対象外。
- テスト時はシークレットウィンドウや別ブラウザで「未ログイン」の状態で公開URLにアクセスすると、キャッシュの影響を減らせる。
